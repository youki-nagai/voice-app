# 0005. 機能追加時にSOLID/DRY品質を維持し一括リファクタリングを防止する

Date: 2026-03-08

## Status

Accepted

## Context

git log分析により、「SOLID/DRY原則に基づくリファクタリング」が4回繰り返されていることが判明した。

- PR #16: `refactor: SOLID/DRY原則に基づく全面リファクタリング`
- PR #56: `refactor: SOLID/DRY原則に基づくコードベース全体のリファクタリング`
- PR #80: `refactor: SOLID/DRY原則に基づくコードベース整理`
- PR #98: `refactor: SOLID/DRY改善 - パネル入力hook抽出、router分割、テスト共通化、デッドコード削除`

パターン: 機能追加フェーズでコードが肥大化 → 一定量溜まると一括リファクタリングで清掃、を周期的に繰り返す「反応型リファクタリング」になっている。

一括リファクタリングは差分が大きくレビューが困難であり、意図しない変更が混入するリスクがある。

## Decision

機能追加時に以下を守り、一括リファクタリングを不要にする:

1. **新しい関数/クラスは単一責任にする** — 1つの関数が複数の責務を持つ場合、実装時点で分割する
2. **既存コードのコピペ禁止** — 重複コードを書こうとしたら、先に共通化してから使う
3. **デッドコードは発生時点で削除** — 機能置き換え時に旧コードを残さない
4. **feat PRに含めてよいリファクタリング** — その機能実装に必要な範囲の構造改善は feat PR に含める。「後でまとめてやる」は禁止

## Consequences

- 各PRの差分に小規模なリファクタリングが含まれるが、一括リファクタリングPRより遥かにレビューしやすい
- 「refactor/solid-dry」ブランチが今後不要になる
- 機能実装の速度がわずかに下がる可能性があるが、一括リファクタリングの工数を考えるとトータルでは削減
