# インライン品質維持

機能追加・バグ修正時に、コード品質をその場で維持する。
「後でまとめてリファクタリング」を禁止し、各PRで品質を完結させる。

背景: ADR 0005 参照。SOLID/DRY一括リファクタリングが4回繰り返された（PR #16, #56, #80, #98）。

## ルール

### 1. 単一責任

新しい関数・クラスを作る際、1つの関数が複数の責務を持たないこと。
責務が2つ以上あるなら、実装時点で分割する。

### 2. 重複禁止

既存コードと同じロジックを書こうとしたら、先に共通化してから使う。
コピペしてから「後で共通化」は禁止。

### 3. デッドコード即時削除

機能を置き換えたら、旧コードをその場で削除する。
「念のため残す」は禁止。

### 4. feat/fix PRにリファクタリングを含める

その機能実装に必要な範囲の構造改善は feat/fix PR に含めてよい。
「refactor/solid-dry」のような一括リファクタリングブランチを作らない。
