# テスト哲学（Google "Software Engineering at Google" 準拠）

ref: https://abseil.io/resources/swe-book/html/ch11.html, https://abseil.io/resources/swe-book/html/ch12.html

## テストの目的

テストの最大の価値は「変更を可能にすること」。バグ発見は副次的効果。

- リファクタリング・機能追加を自信を持って実行するための安全網
- 実行可能なドキュメント（仕様書は陳腐化するが、テストは動く仕様）
- 設計品質の向上（テスト可能な設計 = 疎結合で保守しやすい設計）

## Beyonce Rule

> "If you liked it, then you shoulda put a test on it."

壊れてほしくない振る舞いにはテストを書け。逆に言えば、壊れても困らないものにはテストを書くな。

## テストピラミッド（推奨比率）

```
    5%  E2E / System テスト
   15%  Integration テスト
   80%  Unit テスト
```

アンチパターン:
- アイスクリームコーン型（E2E 多数 + Unit 少数）→ 遅い、不安定、デバッグ困難
- 砂時計型（E2E 多数 + Unit 多数 + Integration 少数）→ コンポーネント間の問題が E2E まで漏れる

## テストサイズ（リソース制約による分類）

| サイズ | プロセス | I/O | ネットワーク | 用途 |
|--------|----------|-----|-------------|------|
| Small | 単一 | 禁止 | 禁止 | ロジック検証。高速・決定的 |
| Medium | 複数可 | 可 | localhost のみ | DB統合、ブラウザ実体化 |
| Large | 複数マシン | 可 | 可 | E2E。ビルド・リリース時のみ実行 |

## 書くべきテスト

### 高価値（積極的に書け）
- ビジネスロジック（services, hooks, utils）
- 状態管理の境界条件（セッション分離、ストリーミング、finalize）
- API 契約（リクエスト/レスポンスの型・ステータスコード）
- 失敗時の動作（例外、エラーハンドリング、タイムアウト）
- 複雑な条件分岐・計算ロジック

### 低価値（書くな）
- UI コンポーネントの存在確認（ボタンがあるか等）
- CSS クラス名・スタイルの検証
- フレームワークの基本動作確認（クリック→コールバック呼ばれるか等）
- E2E で十分カバーされる統合的な画面操作
- getter/setter、単純な委譲メソッド

## ユニットテストの設計原則

### 公開 API 経由でテストしろ
- 実装詳細ではなく公開インターフェースに対してテスト
- テスト破損 = 実際のユーザー影響、を保証する

### ステートテスト > インタラクションテスト
- 良い: `assert accounts.getUser("foo") is not None`
- 悪い: `verify(db).save(any())` → 実装変更で壊れる

### テスト内にロジックを書くな
- 条件分岐・ループ禁止。テストは「見て正しいと分かる」べき
- テストにロジックがあると、テストのテストが必要になる

### Given / When / Then で構造化
```
# Given: 初期状態のセットアップ
# When: テスト対象のアクション実行
# Then: 結果の検証
```

### 命名: 振る舞いを要約
- 良い: `test_given_empty_balance_when_withdraw_then_raises_error`
- 悪い: `test_withdraw`（何を検証しているか不明）
- `and` が入ったら複数の振る舞いをテストしている疑い → 分割

### DAMP > DRY
- テストコードは Descriptive And Meaningful Phrases を優先
- 適度な重複を許容して可読性を確保
- 本番コードの DRY とは異なる基準

### モック・スタブは慎重に
- 実オブジェクトが高速・決定的なら実オブジェクトを使え
- モック乱用は脆弱なインタラクションテストを量産する
- モックを使うなら標準フレームワークに統一

## テスト品質の維持

### 決定性
- フレーキーテスト（非決定的テスト）はテストスイート全体の信頼を破壊する
- 時間依存、スレッドスケジューリング、ネットワーク遅延は排除

### 速度
- 遅いテスト → 実行頻度低下 → 価値低下の悪循環
- sleep/setTimeout の無差別使用禁止。状態遷移のポーリング + タイムアウトを使え

### メンテナンスコスト
- 悪いテストスイートはテストなしより悪い
- テストが壊れて「とりあえずスキップ」が常態化したら赤信号
- テストは本番コードと同等の品質で書け

## 変更に強いテストの基準

理想: 書いた後、システム要件が変わるまで変更不要。

| 変更の種類 | 既存テストへの影響 |
|-----------|------------------|
| リファクタリング | テスト変更なし |
| 新機能追加 | 新テスト追加のみ。既存テスト変更なし |
| バグ修正 | 新テスト追加のみ。既存テスト変更なし |
| 動作変更 | 既存テスト変更が許容される唯一のケース |
